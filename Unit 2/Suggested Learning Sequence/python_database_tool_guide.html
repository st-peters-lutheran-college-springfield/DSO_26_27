Python Database Tool — Build & Test SQL

A lightweight Python + SQLite tool for building and testing your own database before you wrap it in validated code

Lesson 11 onwardPractice scenario shown below
Start here: use this tool to write and run SQL against your own database quickly. It is a testing sandbox, not your final submission — your assessment needs a full Python program with validated INSERT logic (see the last section on this page).

What the Tool Does

  • Enter a name and click Build database to create a new SQLite .db file in the same folder as the tool.
  • Type SQL into the Add SQL box and click Run SQL to execute it against the database you've selected from the dropdown.
  • Click List Tables at any time to check what tables currently exist in your database.
  • SELECT queries are displayed as a neatly formatted table in the Output box; other statements (like CREATE, INSERT, UPDATE) show a short confirmation.

Setup

Step 1

Download the tool

The download contains database_tool.py plus two logo images it needs — keep all three files in the same folder.

Download PythonDatabaseTool.zip
Step 2

Check Python is ready

You need Python 3 with tkinter (already installed on College laptop images). No extra packages required — sqlite3 and tkinter are both part of the standard library.

Step 3

Run it

Unzip the folder, open a terminal inside it, then run:

python database_tool.py

Worked Example — Library Loan System (practice scenario)

This walkthrough uses a small Library Loan System — a different scenario from your devices/events database — so you can see the whole workflow before applying it to your own schema.

Step 1 — Build the database. Type library_practice into the name box and click Build database.

Step 2 — Create the tables. Paste this into the Add SQL box and click Run SQL:

CREATE TABLE books (
    book_id INTEGER PRIMARY KEY AUTOINCREMENT,
    title TEXT NOT NULL,
    author TEXT NOT NULL
);

CREATE TABLE loans (
    loan_id INTEGER PRIMARY KEY AUTOINCREMENT,
    book_id INTEGER NOT NULL,
    member_name TEXT NOT NULL,
    loan_date TEXT NOT NULL,
    FOREIGN KEY (book_id) REFERENCES books(book_id)
);

Step 3 — Confirm the schema. Click List Tables — you should see books and loans in the Output box.

Step 4 — Insert some rows.

INSERT INTO books (title, author) VALUES ('Hyperspace', 'M. Kaku');
INSERT INTO books (title, author) VALUES ('The Martian', 'A. Weir');
INSERT INTO loans (book_id, member_name, loan_date) VALUES (1, 'Priya N.', '2026-08-01');

Step 5 — Query it.

SELECT loans.loan_id, books.title, loans.member_name, loans.loan_date
FROM loans
JOIN books ON loans.book_id = books.book_id;

Run it, read the formatted table in the Output box, then try your own WHERE or GROUP BY version. This same CREATE → INSERT → SELECT pattern is what you'll use to build your devices and events tables for the assessment in Lesson 11.

Moving from the Tool into Your Assessment Code

The tool runs raw SQL directly — it doesn't check whether the data you type is valid before inserting it. Your Part A submission needs more than that: a Python program using the sqlite3 module that validates data before it is inserted (for example, rejecting a null value or an out-of-range classification) and gives the user feedback when it does. Use this tool to get your SQL syntax right and see your table structure working, then move that same SQL into cursor.execute() calls inside a validated Python function — the pattern taught in Lesson 11's direct teaching.

Practise here first with the Library Loan example above — then apply the same CREATE / INSERT / SELECT pattern to your own devices/events schema. Don't reuse the table or field names from this page in your submission; they're deliberately from a different scenario.
