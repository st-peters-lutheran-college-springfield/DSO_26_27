Database Essentials Starter Pack

Vocabulary, a worked example and a hands-on explore-a-database activity — used in Lessons 2, 3 and 11

Week 1 & Week 3Practice scenario — not your assessment data
Start here: every activity on this page uses a sample music database (tracks, downloads, members) — not the devices/events database from your Sunrise Aged Care assessment. Practise the moves here first, then apply them to your own database in Lesson 11.

Why a Different Database?

Learning a new skill on unfamiliar data forces you to actually apply the concept, rather than pattern-matching an answer you've already seen. Once you're confident finding a primary key, spotting redundancy, or running a SELECT query on the music database, you'll do the same thinking — with your own reasoning — on the devices/events database for your assessment.

Check your answers: Table = the whole grid of data for one entity (e.g. all tracks). Field = one column (e.g. Genre). Record = one row (e.g. one track's full details). Q2: TrackID. Q3: "A database that has a single table." Q4: "Each value held must be unique" and "A primary key allows you to identify each unique record" — a primary key does not have to be a number, and it can never repeat.

2. Inefficient Flat File Database — Lesson 3 worked example

The table below stores tracks, downloads and members all in one flat file. Look for the problems this causes before you analyse the redundancy issues in your own provided devices/events database.

TrackIDTitleArtistGenreDloadIDDateTimeMemberIDFirstnameSurname
1Float awayThe SpringsPop14.6.1018:361SaraBibi

Task: highlight every issue you can see with storing the data this way. Look for: data repeated across rows (redundancy), the same fact spelled two different ways ("The Springs" vs "The spings" — an inconsistency an update in one place won't fix everywhere), and how much bigger this table gets every time an existing track is downloaded again. Note down which of these issues are redundancy, and which are integrity risks — you'll need both terms when you write up your storage-comparison justification in Lesson 9.

3. Explore a Real Relational Database (DB Browser for SQLite) — Lesson 11 warm-up, before you build

Before building your own database, open a working one and see how a relational structure actually solves the problems above. The music database has been redesigned into three linked tables: tblTracks, tblDownloads and tblMembers.

Step 1

Get the tools

Download DB Browser for SQLite (free, no login) and the sample database below.

Download DB Browser

Download dbMusic.db

Step 2

Investigate the tables

Open dbMusic.db in DB Browser. On the Database Structure tab, check whether DownlID, TrackID and MemberID in tblDownloads are primary or foreign keys.

Step 3

View the SQL behind a table

Select tblTracks and choose "Modify Table" to see the exact CREATE TABLE SQL that built it — this is what you'll be writing yourself in Lesson 11.

Step 4

Browse and query data

Use Browse Data to explore tblMembers, then go to Execute SQL and run the query below.

Try this query in the Execute SQL tab:

SELECT *
FROM tblDownloads
WHERE date BETWEEN 2011 AND 2013;

Click the play icon to run it, look at what's returned, then modify the year range yourself and re-run it. This is the exact workflow — write SQL, run it, read the result — you'll use with your own database in the next step.
