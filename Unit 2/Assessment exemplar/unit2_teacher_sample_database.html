Teacher Sample — Library Loan Database Build

A worked SQL + Python build at the standard your own devices/events solution should reach

Teacher resourceObjectives 4–7
Note for students: this sample uses a Library Loan System — a different scenario from your Sunrise Aged Care devices/events database. It models the standard and structure expected, not the answer to copy. Table names, field names and logic must be your own.

Schema (3NF)

Relational schema notation for the worked example:

books(book_id, title, author)
members(member_id, first_name, surname, email)
loans(loan_id, book_idFK, member_idFK, loan_date, return_date)

CREATE TABLE SQL

CREATE TABLE books (
    book_id INTEGER PRIMARY KEY AUTOINCREMENT,
    title TEXT NOT NULL,
    author TEXT NOT NULL
);

CREATE TABLE members (
    member_id INTEGER PRIMARY KEY AUTOINCREMENT,
    first_name TEXT NOT NULL,
    surname TEXT NOT NULL,
    email TEXT NOT NULL UNIQUE
);

CREATE TABLE loans (
    loan_id INTEGER PRIMARY KEY AUTOINCREMENT,
    book_id INTEGER NOT NULL,
    member_id INTEGER NOT NULL,
    loan_date TEXT NOT NULL,
    return_date TEXT,
    FOREIGN KEY (book_id) REFERENCES books(book_id),
    FOREIGN KEY (member_id) REFERENCES members(member_id)
);

Validated INSERT (Python + sqlite3)

This is the pattern expected for your insert_event() equivalent — validation happens before the SQL runs, and the function reports back what happened.

import sqlite3
from datetime import datetime

def insert_loan(book_id, member_id, loan_date, db_path="library_practice.db"):
    """Validates and inserts a new loan record. Returns (success, message)."""
    # Validation before anything touches the database
    if not book_id or not member_id:
        return False, "ERROR: book_id and member_id are required."
    try:
        datetime.strptime(loan_date, "%Y-%m-%d")
    except ValueError:
        return False, "ERROR: loan_date must be in YYYY-MM-DD format."

    with sqlite3.connect(db_path) as db:
        cursor = db.cursor()
        # Verify the foreign keys actually exist before inserting
        cursor.execute("SELECT 1 FROM books WHERE book_id = ?", (book_id,))
        if cursor.fetchone() is None:
            return False, f"ERROR: no book with book_id {book_id}."
        cursor.execute("SELECT 1 FROM members WHERE member_id = ?", (member_id,))
        if cursor.fetchone() is None:
            return False, f"ERROR: no member with member_id {member_id}."

        cursor.execute(
            "INSERT INTO loans (book_id, member_id, loan_date) VALUES (?, ?, ?)",
            (book_id, member_id, loan_date)
        )
        db.commit()
        return True, f"Loan inserted for member {member_id}."

Note the use of parameterised queries (? placeholders) rather than string-formatted SQL — this is also the fix for SQL injection covered in Lesson 18.

SELECT / JOIN / GROUP BY — three use-case pattern

Mirrors the "device filter / date filter / summary count" structure your own three queries need to answer for Jordan Kim.

-- 1. Filter: all loans for one member
SELECT * FROM loans WHERE member_id = 2;

-- 2. Join: loan history with readable book titles
SELECT loans.loan_id, books.title, loans.loan_date, loans.return_date
FROM loans
JOIN books ON loans.book_id = books.book_id
WHERE loans.member_id = 2;

-- 3. Summary: loans per member, busiest borrowers first
SELECT member_id, COUNT(*) AS total_loans
FROM loans
GROUP BY member_id
ORDER BY total_loans DESC;

UPDATE / DELETE — preserving referential integrity

-- Safe UPDATE: record a return date
UPDATE loans SET return_date = '2026-08-15' WHERE loan_id = 3;

-- Safe DELETE: remove a test record, not a real loan with dependent history
DELETE FROM loans WHERE loan_id = 99;

Before any DELETE, check what else references that row — deleting a books row that still has active loans pointing to it via book_id would break referential integrity. The same check applies to deleting a device that still has logged events.
