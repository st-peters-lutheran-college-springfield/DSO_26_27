Part B Technical Proposal — Student Scaffold

A planning and writing tool for the Unit 2 Part B proposal (marked out of 25) — use this to plan and structure your own writing, not to copy.

Start here: work through each section below in order. Each one maps directly to a criterion in the Part B ISMG, so completing them all gives you full coverage.

5.1 Problem Analysis and Solution Requirements — Comprehending, Analysing

What it needs: recognise and describe hardware and database principles; symbolise with a DFD, ER diagrams, schema and wireframe; explain interrelationships; analyse the problem and determine success criteria.

Sentence starters:

  • "The ESP32-S3 uses … to …, which matters for this system because …"
  • "The relational schema is in 3NF because …, which avoids the anomaly present in the 2NF version (…)."
  • "Success criterion … will be tested by … and is considered met when …"
Common mistake: describing a diagram instead of explaining why it was designed that way. Always link back to Jordan Kim’s use cases.

5.2 Solution Design — Synthesising

What it needs: a justified comparison of storage approaches, and QCAA-format pseudocode for two modules.

  • "A flat-file approach was rejected because …, whereas the relational approach allows …"
  • Pseudocode should use QCAA-format keywords (BEGIN/END, IF/ELSE, WHILE, INPUT/OUTPUT), not Python syntax.

5.3 Generated Components — Generating

What it needs: two A4 pages of annotated code selected from your Part A work.

  • Choose code that demonstrates the widest range of requirements (CREATE, validated INSERT, SELECT with JOIN/GROUP BY, UPDATE, DELETE, MicroPython, UI).
  • Every annotation should reference a design decision or success criterion, not just describe what the line does.

5.4 Evaluation — Evaluating

What it needs: evaluation against success criteria using test table evidence, a documented refinement, and impact evaluation.

  • "Success criterion … was tested using …, giving an actual result of … against an expected result of … (Pass/Fail)."
  • "Based on this result, I refined … by …, shown by … (before) compared with … (after)."
  • Cover all three impact types — personal (privacy), social (bias) and economic (cost) — with a justified recommendation for each.
Common mistake: listing impacts without evaluating them. "This could raise privacy concerns" is a statement; "no audio is stored, only classification metadata, which mitigates the privacy risk because …" is an evaluation.

5.5 Communication — Communicating

  • Use consistent technical vocabulary throughout — avoid switching between casual and technical terms for the same concept.
  • IEEE reference list: minimum four sources, in-text citation everywhere a source is used.
  • Video: demonstrate the full data path (event → classification → database INSERT → query result) and explain one design decision out loud.
