Teacher Exemplar — Part B Technical Proposal

A model response using the Library Loan System worked example, structured against the same ISMG sections as your own proposal

Teacher resourceModels standard, not content to copy
Note for students: this exemplar is written about the Library Loan System, not the Sunrise Aged Care system you are building. Use it to see what a strong response sounds and looks like at each ISMG criterion — every sentence about the scenario itself must be replaced with your own reasoning about your own system.

5.1 Problem Analysis and Solution Requirements — Comprehending, Analysing

Model paragraph: "The library's existing paper loan register stores each loan as a handwritten card, filed alphabetically by member surname. This creates two constraints: staff cannot quickly answer 'which books does this member currently have out?' without checking every card, and there is no reliable way to prevent a book being logged out to two members at once. The proposed system replaces this with a relational SQLite database (see ER diagram, Fig. 1) accessed through a Python interface, addressing the retrieval-speed constraint directly. The 3NF schema separates books, members and loans into three linked tables, which removes the transitive dependency present in a flat-file version — where a member's name would otherwise repeat on every loan row and risk becoming inconsistent if that name were ever updated (e.g. after a spelling correction)."

Success criterion, written in QCAA/ISMG format: "Loan history retrieval — a query joining loans and books returns the correct set of currently-borrowed titles for a given member_id — pass condition: 100% of 10 test member IDs return the expected row set, verified against the paper register."

What makes this strong: it names a real constraint (concurrent lending), explains why the 3NF design solves it rather than just describing the tables, and states a testable pass condition with a number in it.

5.2 Solution Design — Synthesising

Model paragraph: "A flat-file approach was rejected because member details would repeat on every loan record, creating both wasted storage and an update-anomaly risk (Section 5.1). The relational approach stores each member once and references them by member_id, so a name change requires a single update rather than a search-and-replace across every historical loan."

Pseudocode extract (QCAA format, not Python syntax):

BEGIN insert_loan
  INPUT book_id, member_id, loan_date
  IF book_id NOT FOUND IN books THEN
    OUTPUT "Error: book does not exist"
  ELSE IF member_id NOT FOUND IN members THEN
    OUTPUT "Error: member does not exist"
  ELSE
    INSERT loan record
    OUTPUT "Loan recorded"
  END IF
END insert_loan

5.3 Generated Components — Generating

Model annotation style (for the 2×A4 code pages): rather than "this line inserts a loan," a strong annotation reads: "Parameterised placeholders (?) are used instead of string-formatted values here specifically to prevent SQL injection (Lesson 18) and because two foreign-key existence checks (lines 14–18) are required to meet Success Criterion 2 — a loan cannot be created against a member or book that doesn't exist." Every annotation ties the code back to a design decision or success criterion, never a bare description of syntax.

5.4 Evaluation — Evaluating

Model paragraph: "Success Criterion 2 (loan history retrieval) was tested using the structured test table (Appendix C, Test #2), giving an actual result of three correctly joined rows against an expected result of three rows for member_id 2 (Pass). Test #1 confirmed the validated insert correctly rejected a loan with a missing member_id, returning the expected error message rather than inserting a broken record. Based on early testing, the original design allowed return_date to be required at insert time; this was refined to an optional field, since a loan is created before a book is returned — shown by the original schema (all fields NOT NULL) compared with the revised schema (return_date TEXT, nullable)."

Impact evaluation, all three lenses required:

  • Personal: "Member email addresses are stored for overdue notifications only and are never displayed in the loan-history query output, limiting unnecessary exposure of personal data to staff performing routine lookups."
  • Social: "A classification or sorting rule based on member surname could disadvantage members with non-alphabetic or unusual name formats if not tested against a varied sample — this was mitigated by testing the query against member names with hyphens and non-English characters."
  • Economic: "A commercial library management system was quoted at approximately $40/month; the proposed SQLite-based solution has no ongoing licence cost, though it requires volunteer/staff time to maintain, which is a fair trade-off recommendation for a small school library's budget."
What makes this strong: each impact is evaluated with evidence and a judgement, not just named. "This could raise privacy concerns" alone would not meet this standard.

5.5 Communication — Communicating

Technical vocabulary is used consistently throughout (always "foreign key," never switching to "linking field" partway through). The reference list below shows the expected IEEE format for two source types:
