Why light matters for Daniel

👤 Daniel's situation

Daniel is a shift worker who sleeps at irregular times — sometimes in full daylight. A bright room signals "daytime" to the brain and suppresses melatonin, making it harder to sleep even when tired.

💡 What the LDR measures

An LDR (Light Dependent Resistor) changes its resistance based on how much light falls on it. The microcontroller reads this as an analogue voltage — a number from 0 to 65535. Lower = darker. Higher = brighter.

SC6 Success criterion

"Low light level = suitable for rest." Your algorithm must check the LDR value against a threshold and include light in the combined rest score — alongside motion, temperature, and humidity.

⚠️

LDR values vary — calibration is part of the task

Unlike temperature, there is no fixed "correct" threshold for light. A value of 8000 might mean dark in one room and dim in another, depending on the sensor, the room, and the time of day. Part of your FIA2 journal is documenting the threshold you chose and why — this is evidence for your Evaluating criterion.

🔥 FireBeetle 2 ESP32-E pathway
✅

Gravity LDR → IO Shield A0 port → GPIO36

The Gravity Ambient Light Sensor uses a 3-pin Gravity cable — same type as your PIR and DHT sensors. It plugs into any A (analogue) port on the IO Shield. Use A0, which maps to GPIO36 on the ESP32. GPIO36 is an ADC1 pin — the correct ADC channel for MicroPython on the ESP32-E.

Step 1 — Identify your LDR sensor
🔍

Look for the Gravity Ambient Light Sensor in your kit

It is a small PCB module with a light-sensitive window and a 3-pin Gravity connector. It may be labelled "Gravity: Ambient Light Sensor", "BH1750", or simply have a photoresistor component visible. It uses the same 3-pin cable (black GND, red VCC, yellow SIG) as the PIR and DHT sensors.

If your kit has a Gravity Analog Ambient Light Sensor (resistive LDR) — use it on the A0 port. If it has a digital BH1750 (4-pin I²C) — use the I²C port instead (see the note below).

Step 2 — Wire the LDR to the IO Shield

Your PIR (D3) and DHT sensor (D2 or I²C) stay plugged in exactly as they were in Lesson 2. You are adding one more sensor today.

Red VCC A0 port — VCC pin 3.3V 3.3V power — do NOT use 5V
Yellow A0 → GPIO36 A0 port — SIG pin GPIO36 (ADC1_CH0) Analogue signal — ADC reads 0–4095 on ESP32
⚠️

ESP32 ADC — important difference from the Pico W

The ESP32 ADC returns values from 0 to 4095 (12-bit). The Pico W ADC returns 0 to 65535 (16-bit). Your threshold values in code must match your board. The ESP32 A0 port uses ADC(Pin(36)) — do not use GPIO34 or GPIO35 which are input-only pins.

if SENSOR in ('DHT11', 'DHT22'):
    import dht
    _dht = dht.DHT11(Pin(25)) if SENSOR == 'DHT11' else dht.DHT22(Pin(25))
    def read_dht():
        try:
            _dht.measure()
            return _dht.temperature(), _dht.humidity()
        except:
            return None, None
else:   # DHT20 via I²C
    from machine import I2C
    _i2c = I2C(0, scl=Pin(22), sda=Pin(21))
    def read_dht():
        try:
            _i2c.writeto(0x38, bytes([0xAC, 0x33, 0x00]))
            time.sleep_ms(80)
            d = _i2c.readfrom(0x38, 7)
            hum  = ((d[1] << 12) | (d[2] << 4) | (d[3] >> 4)) / 1048576.0 * 100
            temp = (((d[3] & 0x0F) << 16) | (d[4] << 8) | d[5]) / 1048576.0 * 200 - 50
            return round(temp, 1), round(hum, 1)
        except:
            return None, None

def read_sensors():
    global last_motion_ms

    # PIR
    if pir.value() == 1:
        last_motion_ms = time.ticks_ms()
    elapsed_ms = time.ticks_diff(time.ticks_ms(), last_motion_ms)
    calm_secs  = elapsed_ms // 1000
    motion_now = pir.value() == 1

    # LDR — NEW
    ldr_raw  = ldr.read()                      # 0–4095 on ESP32
    light_ok = ldr_raw < LIGHT_THRESHOLD       # True = dark enough

    # DHT
    temp, hum = read_dht()

    return calm_secs, motion_now, ldr_raw, light_ok, temp, hum

def evaluate_rest(calm_secs, motion_now, light_ok, temp, hum):
    issues = []

    # SC7 — motion check
    motion_ok = (not motion_now) and calm_secs >= CALM_SECONDS
    if motion_now:       issues.append('Motion detected')
    elif not motion_ok:  issues.append('Settling — {}s of {}s'.format(calm_secs, CALM_SECONDS))

    # SC6 — light check (NEW)
    if not light_ok:     issues.append('Room too bright')

    # SC5 — temperature check
    temp_ok = temp is not None and 16 <= temp <= 22
    if temp is None:     issues.append('Temp sensor error')
    elif temp < 16:     issues.append('Too cold ({:.1f}°C)'.format(temp))
    elif temp > 22:     issues.append('Too warm ({:.1f}°C)'.format(temp))

    # Humidity check
    hum_ok = hum is not None and hum <= 70
    if hum is not None and not hum_ok:
        issues.append('Humidity high ({:.0f}%)'.format(hum))

    all_ok = motion_ok and light_ok and temp_ok and hum_ok

    if all_ok:
        return 'SUITABLE FOR REST', '\u2713', 'All conditions good'
    elif len(issues) == 1:
        return 'MONITOR CONDITIONS', '~', issues[0]
    else:
        return 'NOT SUITABLE', '\u25cf', ' · '.join(issues)

🍓 Raspberry Pi Pico W pathway — Mac M4
✅

Gravity LDR → direct jumper wires → GP26 (ADC0)

The Pico W has no IO Shield, so the LDR connects via female-to-male Dupont jumper wires directly to the Pico W pins — the same way your DHT22 and PIR are already connected. The LDR signal wire goes to GP26, which is ADC channel 0. This is already in your existing Pico W code.

⚠️

GP26 is already used for the LDR in your existing Pico W code

Looking back at the main.py from the Pico W setup guide, ldr = ADC(Pin(26)) was already included. If you have been running that code without the LDR connected, the ldr.read_u16() call still works — it just reads floating/noise values. Today you physically connect the sensor to make SC6 functional.

Step 1 — Wire the LDR to the Pico W

Your PIR (GP14) and DHT22 (GP15) stay connected as before. You are adding a third sensor.

Teacher notes
👩‍🏫

Before the lesson — checklist

  • Confirm which students have a Gravity Analog LDR vs a digital BH1750 — wiring differs
  • Mac M4 students (billyGoat pathway) already have ldr = ADC(Pin(26)) in their code — just need to physically wire the sensor and set the threshold
  • Key teaching pause: "Why is there no fixed correct threshold for light, but there is a fixed correct threshold for temperature?" — connects to the difference between sensor calibration (device-dependent) and physiological research (evidence-based)
  • Require students to record two calibration readings in their journal before writing the threshold into code — this turns calibration into assessment evidence
  • Common issue on ESP32: forgetting ldr.atten(ADC.ATTN_11DB) — without it the ADC reads a compressed range and all values cluster near 0
  • Pico W note: read_u16() not read() — students may copy ESP32 code by mistake
⚠️

If students have a BH1750 digital light sensor (I²C, 4-pin) instead of the analog LDR

The BH1750 uses I²C at address 0x23 and returns lux values directly (no calibration needed). For ESP32: connect to the I²C port (SDA=GPIO21, SCL=GPIO22). The MicroPython driver is a short inline class — no library install needed. Ask Ms Baessler for the BH1750 driver code snippet if you have this sensor. The SC6 logic is simpler: lux < 50 = suitable for rest (50 lux is dim indoor lighting).
