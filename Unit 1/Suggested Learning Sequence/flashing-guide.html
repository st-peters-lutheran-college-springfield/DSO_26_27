Step 0 — The cable problem (read first)

Not all USB-C cables carry data

The FireBeetle 2 uses USB-C. USB-C cables look identical from the outside but behave very differently inside. Charge-only cables contain just two power wires — no data lines — so your computer will never see the board no matter how many times you retry.

⚡ Charge-only cable

  • Powers the board (LED lights up)
  • No COM port appears in Device Manager
  • Came with headphones, power banks, cheap phone accessories
  • Often thinner and lighter
✗ Will not work for flashing

🔄 Data + charge cable

  • Powers the board AND creates a COM / tty port
  • Labelled USB 2.0, USB 3.2, or "data cable"
  • Came with Android phones, printers, Arduino kits
  • Usually slightly thicker, stiffer
✓ Use this one
⚠
Quick test: Plug in and check Device Manager (Windows) or run ls /dev/tty.* (Mac/Linux). If nothing new appears, swap the cable before doing anything else. This is the single most common cause of "board not found" errors.

Reference: newell.pro — Charging vs data transfer in USB-C cables

Step 1 — Install the CH340 driver

The board uses a CH340 USB chip

The FireBeetle 2's USB-to-serial bridge is a CH340. Your OS needs a driver to talk to it. Without it, no COM port appears even with a good data cable.

Windows

Windows 10 / 11

  1. Plug the board in with a data cable. Open Device Manager (Win + X → Device Manager).

  2. Look under Ports (COM & LPT). If you see CH340 — you already have the driver, skip to Step 2. If you see a yellow ! under "Other devices", continue.

  3. Download the CH340 driver from the DFRobot wiki: wiki.dfrobot.com/dfr0654 → click Driver in the downloads list.

  4. Extract the zip, run SETUP.EXE, click Install. Unplug and replug the board. A COM port (e.g. COM22) should now appear.

ℹ
If COM22 appears but you still can't flash, try COM3, COM4 — Windows sometimes assigns a different number after driver install. The number does not matter as long as it appears.
macOS

macOS 12 Monterey and later

  1. macOS includes a CH340 driver from version 12 onwards. Plug in the board and open Terminal.

  2. Run: ls /dev/tty.* — you should see /dev/tty.usbserial-XXXX or /dev/tty.wchusbserial.

  3. If nothing appears, download the macOS CH340 driver from the same DFRobot driver link above and install the .pkg file. Restart if prompted.

  4. After install, you may need to approve the kernel extension in System Settings → Privacy & Security.

Linux

Linux (Ubuntu / Debian)

  1. The CH340 driver (ch341) is built into the Linux kernel — no install needed.

  2. Plug in the board and run: dmesg | grep tty — you should see /dev/ttyUSB0.

  3. If Thonny says "permission denied", add yourself to the dialout group and log out/in:

    sudo usermod -aG dialout $USER
Step 2 — Flash MicroPython firmware

This only needs to happen once per board

MicroPython firmware replaces the factory firmware. Once it is on the board, you just upload .py files — no reflashing needed unless something goes badly wrong.

✓
Already done? If Thonny connects and you see a >>> Python prompt at the bottom, MicroPython is already installed. Skip to Step 3.
All platforms

Flash using Thonny (easiest method)

  1. Download and install Thonny from thonny.org — available for Windows, macOS, and Linux.

  2. Open Thonny. Go to Tools → Options → Interpreter.

  3. From the interpreter dropdown select MicroPython (ESP32).

  4. Click Install or update MicroPython (the link at the bottom of the dialog).

  5. Set Target port to your COM / tty port. Set variant to ESP32 / WROOM. Click Install.

  6. Thonny downloads and flashes the firmware automatically. Takes about 30 seconds.

⚠
If the flash stalls at "Connecting…" — see Boot Fix below before retrying.
Advanced

Flash manually with esptool (command line)

Use this if Thonny's built-in flasher fails, or you prefer the command line.

  1. Install esptool: pip install esptool

  2. Download the firmware .bin from micropython.org/download/ESP32_GENERIC. Choose the latest stable release.

  3. Erase the flash first (replace COM22 / ttyUSB0 with your port):

    # Windows
esptool.py --port COM22 --baud 460800 erase_flash

# macOS / Linux
esptool.py --port /dev/ttyUSB0 --baud 460800 erase_flash

  4. Flash the firmware:

    esptool.py --port COM22 --baud 460800 \
  write_flash -z 0x0 ESP32_GENERIC-20241129-v1.28.0.bin

Full reference: docs.micropython.org — Deploying the firmware

Step 3 — Configure Thonny

Tell Thonny where your board is

All platforms

One-time Thonny setup

  1. Open Thonny. Menu: Tools → Options → Interpreter.

  2. Interpreter: MicroPython (ESP32).

  3. Port: select your COM port (Windows) or /dev/ttyUSB0 / /dev/tty.usbserial-XXXX (Mac/Linux). If unsure, unplug the board, check what ports are listed, plug back in, and pick the new one.

  4. Click OK. The bottom panel should show a >>> prompt. If it does, MicroPython is running. ✓

Step 4 — Upload your project files

Sending main.py to the board

For this project you need one file on the board: main.py. MicroPython runs it automatically every time the board powers on.

All platforms

Upload via Thonny file panel

  1. In Thonny, open main.py from your computer (File → Open).

  2. Make sure the bottom REPL panel shows >>> (board is connected).

  3. Go to File → Save copy… → MicroPython device. Save the file as main.py.

  4. Press the red Stop button, then the green Run button (or press RST on the board). The REPL should print your hotspot IP address.

ℹ
The Files panel (View → Files) shows both your computer and the board side-by-side. You can drag files across or right-click → Upload to / Download to.
Advanced

Upload via mpremote (command line)

# Install
pip install mpremote

# Copy main.py to the board
mpremote connect COM22 cp main.py :

# Soft-reset to run it immediately
mpremote connect COM22 reset
Troubleshooting — Wrong boot mode (0x13 error)

Board not entering download mode

The error Wrong boot mode detected (0x13) means the ESP32 booted into run mode instead of download mode. The chip needs GPIO0 held LOW during reset to accept a flash. Here is the fix sequence in order of effort.

!
Try cable swap first. A charge-only cable can cause this exact error even if the board appears to power on. Swap the cable before attempting anything below.

Fix 1 — Manual button timing (works 90% of the time)

The BOOT button is the right button on the FireBeetle 2. RST is the left button.

Hold BOOT → Tap RST once → Release BOOT → Click Install in Thonny

Do this immediately when Thonny starts connecting. You have about a 2-second window.

Fix 2 — Lower the baud rate

High baud rates can fail on some USB hubs and cables. Drop to 115200:

esptool.py --port COM22 --baud 115200 \
  --before default_reset --after hard_reset \
  write_flash -z 0x0 ESP32_GENERIC-v1.28.0.bin

The --before default_reset flag makes esptool toggle RTS/DTR to trigger boot mode — no button pressing needed.

Fix 3 — Bypass the BOOT button with a jumper wire

The BOOT button simply shorts IO0 to GND. You can do this manually:

  1. Locate the IO0 and GND pins on the board header (check the DFRobot pinout diagram).

  2. Hold a jumper wire or metal object bridging IO0 to GND.

  3. Press and release RST.

  4. Remove the wire — flash immediately.

Fix 4 — Plug directly into the laptop

USB hubs — especially unpowered ones — do not supply enough current for the ESP32 during flashing. Plug the board directly into a laptop or desktop USB port, not a hub or charger.

Step 5 — Verify it worked

Quick sensor test before the full project

Paste this into Thonny's REPL and press Enter. It reads the PIR and DHT22 directly — no web server needed.

# Paste into Thonny REPL — tests PIR + DHT22
from machine import Pin
import dht, time

pir = Pin(26, Pin.IN)
d   = dht.DHT22(Pin(25))

for _ in range(6):
    d.measure()
    print('PIR:{} | Temp:{:.1f}C | Hum:{:.0f}%'.format(
          pir.value(), d.temperature(), d.humidity()))
    time.sleep(2)
✓
You should see six lines of readings. PIR shows 0 at rest and 1 when you wave. If temp/hum shows numbers — you're ready to run main.py.
