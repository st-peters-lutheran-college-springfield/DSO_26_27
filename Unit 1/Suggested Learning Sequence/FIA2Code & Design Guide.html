The four CARP principles

CARP stands for Contrast, Alignment, Repetition, Proximity. These are the core visual communication principles from your textbook (Table 3.2.2). For Daniel, Contrast is the most critical because of his colour-vision deficiency.

C
Contrast
Use markedly different colours, sizes or weights to emphasise elements. High contrast improves readability — especially for colour-vision deficiency and low-light conditions.
Daniel's interface: Dark text (#0D1B2A) on light background (#F5F8FC). Status text ≥48px bold. WCAG ratio ≥4.5:1. Status never relies on colour alone — always paired with a symbol and text label.
A
Alignment
Align elements to create visual order and connections. Clean alignment aids readability and shows connections between different elements.
Daniel's interface: All content centred on a single column. Status block, sensor readings, and timestamp stacked vertically — clear reading order top to bottom. No sidebar or columns to scan.
R
Repetition
Repeat visual elements (colours, shapes, icons, fonts) throughout the design for consistency. Reduces cognitive load by making the interface predictable.
Daniel's interface: Same font (Arial), same status box shape, same icon set used consistently. Three statuses (SUITABLE / MONITOR / UNSUITABLE) always appear in the same position and format.
P
Proximity
Position related elements close together to show they belong to a group. Improves learnability and reduces visual clutter.
Daniel's interface: All sensor readings (temp, light, motion) grouped in one card. The primary status is visually separated from raw data — Daniel sees the verdict first, details second.

WCAG 2 — WebAIM checklist applied to Daniel

WCAG 2 is organised around four principles: Perceivable, Operable, Understandable, Robust (POUR). The most relevant for Daniel are Perceivable (colour, contrast) and Understandable (clear language, status words).

WCAG PrincipleGuidelineHow to apply for DanielHow to test
Perceivable 1.1 Text alternatives Every icon must have a text label. Don't use ✅ alone — write "SUITABLE ✅" so screen readers and colour-blind users both understand it. Remove all CSS — does the meaning still come through as plain text?
Perceivable 1.3 Colour is not the only signal The status box background colour (red/amber/green) must also show different text labels and symbols. Print the page in greyscale — is each status still distinguishable? Screenshot → convert to greyscale in any image editor → can you read each status?
Perceivable 1.4.3 Contrast ratio ≥ 4.5:1 Check all text against its background using the WebAIM Contrast Checker. SC4 requires ≥4.5:1 for primary text. webaim.org/resources/contrastchecker
Understandable 3.1 Readable language Use plain English status words: "SUITABLE FOR REST", "MONITOR CONDITIONS", "NOT SUITABLE". Avoid jargon or abbreviations. Ask: if Daniel checks this at 6:30am after a night shift, can he understand it in under 3 seconds?
Understandable 3.3 Error identification If the sensor disconnects, show "SENSOR OFFLINE — last update X seconds ago". Never show no status at all — Daniel needs to know if the data is stale (SC8). Disconnect the sensor board. Does the interface show a clear error state within 15 seconds?
Robust 4.1 Compatible with assistive technology Use semantic HTML elements. <main>, <h1>, <p> — not just <div> for everything. Validate HTML at validator.w3.org

Option A — Single Pico W (simpler, recommended for most students)

All sensors connect directly to the Pico W GPIO pins. The Pico W runs both the sensor reading code and the web server. Students connect their tablet to the Pico W's WiFi hotspot.

Sensor + server structure (Pico W)

main.py — Pico W single-board version
# ============================================================
#  Home Rest & Recovery Monitor — Pico W single-board version
#  Sensors: DHT22 (GP13)  LDR → ADC (GP26)  PIR (GP14)
# ============================================================
import network, socket, dht, time
from machine import Pin, ADC

# ── WiFi hotspot settings ────────────────────────────────
AP_SSID     = 'RestMonitor'
AP_PASSWORD = 'rest1234'

ap = network.WLAN(network.AP_IF)
ap.active(True)
ap.config(essid=AP_SSID, password=AP_PASSWORD)
while not ap.active():
    time.sleep(0.5)
ip = ap.ifconfig()[0]   # Always 192.168.4.1
print("Hotspot ready: connect to", AP_SSID)
print("Open browser:  http://", ip)

# ── Hardware setup ───────────────────────────────────────
dht_sensor = dht.DHT22(Pin(13))    # Temperature + humidity
ldr        = ADC(Pin(26))           # Light-dependent resistor
pir        = Pin(14, Pin.IN)        # Passive infrared motion

# Motion timer — tracks how long since last motion
last_motion_time = time.time()

# ── Classification algorithm ─────────────────────────────
def classify_environment(temp, hum, ldr_val, motion_clear):
    # Returns: label, symbol, bg_color, detail
    # Addresses SC1, SC3, SC5, SC6, SC7
    reasons = []

    # SC5 — temperature check
    temp_ok = 18 <= temp <= 22
    if temp < 18: reasons.append("Too cold")
    if temp > 22: reasons.append("Too warm")

    # SC6 — light check (LDR: lower value = brighter; threshold tested)
    light_ok = ldr_val < 20000  # adjust threshold for your LDR
    if not light_ok: reasons.append("Too bright")

    # SC7 — motion check
    motion_ok = motion_clear
    if not motion_ok: reasons.append("Motion detected")

    # Classify overall status
    all_ok = temp_ok and light_ok and motion_ok

    if all_ok:
        return "SUITABLE FOR REST", "✅", "#1B4332", "Conditions are good"
    elif len(reasons) == 1:
        return "MONITOR CONDITIONS", "🟡", "#7A5000", reasons[0]
    else:
        return "NOT SUITABLE FOR REST", "⚠️", "#2C1A0E", ", ".join(reasons)

# ── Web page generator ───────────────────────────────────
def build_page(temp, hum, ldr_val, motion_clear, last_update):
    label, symbol, bg_color, detail = classify_environment(
        temp, hum, ldr_val, motion_clear)

    # SC3: sensor readings shown BELOW status, not inside it
    # SC4: status text 56px bold
    # SC1: symbol + text always together
    # SC2: auto-refresh every 5 seconds
    return f"""<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <meta http-equiv="refresh" content="5">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>Rest Monitor</title>
  <style>
    body {{ font-family: Arial, sans-serif; background: #F5F8FC;
            margin: 0; padding: 20px; text-align: center; }}
    .status-box {{ background: {bg_color}; color: #FFFFFF;
                   border-radius: 16px; padding: 30px 20px;
                   margin: 20px auto; max-width: 480px; }}
    .status-symbol {{ font-size: 64px; display: block; margin-bottom: 10px; }}
    .status-label  {{ font-size: 52px; font-weight: 700; display: block; }}
    .status-detail {{ font-size: 20px; margin-top: 12px; opacity: 0.9; }}
    .readings {{ background: #FFFFFF; border-radius: 12px;
                 padding: 16px; margin: 16px auto; max-width: 480px;
                 border: 1px solid #C4D8EE; font-size: 18px; color: #0D1B2A; }}
    .readings p {{ margin: 8px 0; }}
    .timestamp {{ font-size: 14px; color: #3D5A78; margin-top: 12px; }}
  </style>
</head>
<body>
  <h1 style="font-size:24px; color:#0D1B2A;">Home Rest Monitor</h1>

  <div class="status-box">
    <span class="status-symbol">{symbol}</span>
    <span class="status-label">{label}</span>
    <div class="status-detail">{detail}</div>
  </div>

  <div class="readings">
    <p>🌡️ Temperature: <strong>{temp:.1f} °C</strong></p>
    <p>💧 Humidity: <strong>{hum:.0f} %</strong></p>
    <p>💡 Light: <strong>{"Low" if ldr_val < 20000 else "High"}</strong></p>
    <p>👁️ Motion: <strong>{"None detected" if motion_clear else "Detected"}</strong></p>
  </div>

  <p class="timestamp">Last updated: {last_update}</p>
  <p class="timestamp">Auto-refreshes every 5 seconds</p>
</body></html>"""

# ── Web server setup ─────────────────────────────────────
addr   = socket.getaddrinfo('0.0.0.0', 80)[0][-1]
server = socket.socket()
server.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)
server.bind(addr)
server.listen(1)
server.settimeout(2)
print("Server running at http://", ip)

# ── Main loop ────────────────────────────────────────────
while True:
    try:
        # Read sensors
        dht_sensor.measure()
        temp    = dht_sensor.temperature()
        hum     = dht_sensor.humidity()
        ldr_val = ldr.read_u16()   # 0–65535

        # SC7 — update motion timer
        if pir.value() == 1:          # motion detected
            last_motion_time = time.time()
        motion_clear = (time.time() - last_motion_time) >= 30

        timestamp = "{:02d}:{:02d}:{:02d}".format(
            *time.localtime()[3:6])

        try:
            client, addr = server.accept()
        except:
            continue

        client.recv(1024)  # discard request — page is read-only

        page = build_page(temp, hum, ldr_val, motion_clear, timestamp)
        client.send("HTTP/1.1 200 OK\r\n"
                    "Content-Type: text/html; charset=UTF-8\r\n\r\n")
        client.send(page)
        client.close()

    except Exception as e:
        print("Error:", e)
        time.sleep(2)

SC8 — sensor offline / stale data detection

If the sensor board stops sending data, Daniel must see a clear error — not stale readings with no warning. Add this to your classification logic:

SC8 — stale data flag
# Track time of last successful sensor read
last_good_read = time.time()

# Inside your main loop, after sensor.measure():
try:
    dht_sensor.measure()
    temp = dht_sensor.temperature()
    hum  = dht_sensor.humidity()
    last_good_read = time.time()    # update timestamp on success
except:
    # sensor failed — check if data is stale (SC8: 15 second threshold)
    age = time.time() - last_good_read
    if age > 15:
        # Show error state in the page
        page = build_error_page(age)
        client.send("HTTP/1.1 200 OK\r\n"
                    "Content-Type: text/html; charset=UTF-8\r\n\r\n")
        client.send(page)
        client.close()
        continue

def build_error_page(age_seconds):
    return f"""<!DOCTYPE html><html><head>
  <meta charset="UTF-8">
  <meta http-equiv="refresh" content="5">
  <style>
    body{{font-family:Arial;background:#1C1C1C;color:#F5F8FC;
         text-align:center;padding:40px;}}
    .offline-box{{background:#3D5A78;border-radius:16px;
                  padding:30px;max-width:480px;margin:0 auto;}}
    .offline-symbol{{font-size:64px;display:block;}}
    .offline-label{{font-size:48px;font-weight:700;display:block;margin-top:10px;}}
  </style></head><body>
  <div class="offline-box">
    <span class="offline-symbol">📡</span>
    <span class="offline-label">SENSOR OFFLINE</span>
    <p style="font-size:20px;margin-top:16px;">
      Last data received {age_seconds:.0f} seconds ago
    </p>
    <p style="font-size:16px;opacity:0.8;">Check sensor connection</p>
  </div>
</body></html>"""

Code annotation guide — how to annotate for your journal

Your journal requires 2 × A4 pages of annotated code. Each annotation must link code to a design decision, success criterion, or usability principle. Here are the three annotation types markers expect:

motion_clear = (time.time() - last_motion_time) >= 30
Type 1 — Algorithm annotation (SC7) This Boolean expression evaluates SC7: it returns True only when 30 or more seconds have elapsed since the last PIR trigger. time.time() returns seconds since boot (monotonic). The ≥30 threshold was determined by desk-checking three test cases. Used in classify_environment() to contribute to the overall status.
.status-label {{ font-size: 52px; font-weight: 700; }}
Type 2 — Design annotation (SC4 + CARP) 52px exceeds the SC4 minimum of 48px. font-weight 700 adds size contrast (CARP: Contrast). Together these allow Daniel to read the status from 2 metres on an older tablet. Contrast ratio for white text on #1B4332 is 8.1:1, exceeding WCAG 2.0 Level AA (4.5:1).
return "SUITABLE FOR REST", "✅", "#1B4332", detail
Type 3 — Usability annotation (SC1 + SC3) Returns a text label ("SUITABLE FOR REST") paired with a symbol ("✅") to satisfy SC1 — status is never communicated by colour alone. The label contains no numbers (SC3). The bg_color parameter is used for supplementary colour reinforcement only, not as the sole signal.

📌 Raspberry Pi Pico W — GPIO Pinout Reference

Option 1 — SVG circular gauge (like a speedometer)

An SVG <circle> with stroke-dasharray creates a gauge arc. You inject the temperature value from MicroPython into the stroke-dashoffset to move the needle. This works perfectly served from a Pico W.

temperature_gauge.html — SVG arc gauge snippet
<svg viewBox="0 0 120 120" width="200" xmlns="http://www.w3.org/2000/svg">
  <!-- Background ring -->
  <circle cx="60" cy="60" r="50"
    fill="none" stroke="#1A3A5C" stroke-width="10"/>
  <!-- Value arc: stroke-dasharray=circumference(~314), stroke-dashoffset controls fill -->
  <!-- MicroPython injects: offset = 314 - (temp/40*314) -->
  <circle cx="60" cy="60" r="50"
    fill="none" stroke="#2B8BE0" stroke-width="10"
    stroke-dasharray="314"
    stroke-dashoffset="{offset}"
    transform="rotate(-90 60 60)"/>
  <!-- Centre label -->
  <text x="60" y="55" text-anchor="middle"
    font-size="22" fill="#D6E8FB" font-weight="bold">{temp:.1f}</text>
  <text x="60" y="72" text-anchor="middle"
    font-size="11" fill="#5A7A9A">°C</text>
</svg>

🖥 Running two Thonny windows on one Mac — two devices simultaneously

By default macOS treats Thonny as a single-instance application — clicking the icon a second time just brings the existing window to the front. But you can force macOS to open a completely separate process using the open -n Terminal command. Each instance holds its own serial port connection independently.

⚠️ Why this matters — port locking

Each serial port (/dev/cu.usbmodem…) can only be held by one process at a time. If you try to connect a second Thonny window to a port already held by the first window, you will see: "Could not exclusively lock port [Errno 35] Resource temporarily unavailable". The solution is separate Thonny processes — not just separate windows of the same process.

Step-by-step: open a second (or third) Thonny instance on Mac

1 Connect your first device via USB Plug in normally — no BOOTSEL held
2 Open Thonny normally (from Applications or Dock) This is instance 1 — connect it to the first device in Run → Configure Interpreter
3 Connect your second device via USB Each device gets its own port number: usbmodem1101, usbmodem1201 etc.
4 Open Terminal (Spotlight → type Terminal → Enter) Terminal is in Applications → Utilities
5 Type the command below and press Enter The -n flag forces a new process — macOS launches a completely separate Thonny
6 In the second Thonny, go to Run → Configure Interpreter → select the second port Choose the port not already in use by instance 1

📛 Naming conventions on microcontrollers — why this is critical

Microcontrollers are not like a laptop. There is no operating system managing multiple user files, no "open with" dialogue, and no file browser. When a Pico W or ESP32 powers on, it looks for exactly two filenames in its root directory: boot.py and main.py. These names are hardwired into MicroPython. No other filenames will run automatically.

🚨 The problem — students saving files like "billyGoatboot.py" or "billyGoatmain.py"

If a student saves their startup code as billyGoatboot.py instead of boot.py, the Pico W will do nothing when powered on. The file exists on the board's flash storage but MicroPython will never execute it automatically. This is a silent failure — Thonny won't warn you, the board won't error — it simply runs nothing.

This catches students out constantly because on a laptop, you can name files anything you like. On a microcontroller, the filename is the instruction to the operating system.

What the two reserved filenames actually do

FilenameWhen it runsWhat to put in itWhat NOT to put in it
boot.py Runs first, every time the board powers on or resets — before main.py Hardware initialisation, WiFi connection, setting up pins that must be ready before anything else Your main application loop — if boot.py crashes, main.py never runs
main.py Runs second, automatically after boot.py completes Your main application — sensor reading loop, web server, classification algorithm Anything that only needs to run once at setup — put that in boot.py
Any other name
(e.g. test.py, sensor.py)		 Never runs automatically — must be explicitly imported or run from Thonny Helper modules you import from main.py — e.g. import sensor Your main code if you want it to run without a laptop connected
⚠️ The naming convention rule for this project

Your filename convention for keeping track of your own files on your laptop (OneNote, Downloads folder, etc.) is sensible — billyGoat_main_v2.py is a great filename for a backup copy on your Mac. But when you save to the MicroPython device via Thonny (File → Save as → MicroPython device), the file must be named main.py — no prefix, no suffix, no version number.

📡 How IoT boards handle networking and WiFi

IoT stands for Internet of Things — physical devices connected to a network to send or receive data. Understanding how the Pico W and ESP32 handle networking is essential for FIA2, because Daniel's interface is delivered over WiFi.

The two WiFi modes — AP vs STA

ModeNameWhat the board doesWho connects to whomUse case for Daniel
AP Access Point mode The board creates its own WiFi network (like a router). It broadcasts an SSID you can see in your WiFi list. Daniel's tablet connects TO the board's network. Board IP is always 192.168.4.1. Recommended for FIA2 — works anywhere, no external router needed. Daniel just connects to "RestMonitor".
STA Station mode The board joins an existing WiFi network (like your phone or home router). Both the board and Daniel's tablet are on the same network. Board gets a dynamic IP from the router. More realistic for a real deployment but harder to demo — IP address changes. Not recommended for FIA2 unless you set a static IP.
ap_vs_sta_mode.py — code comparison
import network, time

# ── AP MODE (Access Point) — board creates its own WiFi ──────
ap = network.WLAN(network.AP_IF)
ap.active(True)
ap.config(essid='RestMonitor', password='rest1234')
while not ap.active():
    time.sleep(0.5)
print('AP mode — connect to RestMonitor')
print('Open browser: http://192.168.4.1')  # IP is always this in AP mode

# ── STA MODE (Station) — board joins existing WiFi ───────────
sta = network.WLAN(network.STA_IF)
sta.active(True)
sta.connect('YOUR_WIFI_SSID', 'YOUR_WIFI_PASSWORD')
while not sta.isconnected():
    time.sleep(0.5)
ip = sta.ifconfig()[0]   # IP assigned by router — different each time
print('STA mode — connected, IP:', ip)
print('Open browser: http://' + ip)

How a request travels from Daniel's tablet to the Pico W

🔁 The request-response cycle — what happens every 5 seconds

Understanding this explains why <meta http-equiv="refresh" content="5"> gives Daniel live data, and why SC2 works:

  1. Daniel's browser sends an HTTP GET request to http://192.168.4.1 every 5 seconds (triggered by the meta refresh tag)
  2. The Pico W's web server (srv.accept() in your while True loop) receives the connection
  3. MicroPython reads the sensors right at that moment — DHT22, LDR, PIR
  4. The classification algorithm runs: temperature + light + motion → SUITABLE / MONITOR / UNSUITABLE
  5. The Pico W builds the HTML page with current values injected via .format()
  6. The HTML is sent back as an HTTP response — Daniel's browser displays it

This is called a synchronous poll model — the page is rebuilt from scratch on every request. It is the simplest possible IoT web server pattern and perfectly suited to MicroPython on a Pico W.

🔗 Can you link an external CSS file from a Pico W?

Yes — if you serve it from the Pico W itself. You can save both dashboard.html and styles.css to the Pico W's flash storage and serve the CSS file as a separate HTTP response when the browser requests it. This requires a slightly more complex server that checks the request URL and serves the correct file. For most FIA2 students, inlining the CSS (pasting it into a <style> block in the HTML) is simpler and perfectly acceptable. The separate file approach is documented below for students who want to extend their solution.

multi_file_server.py — serving HTML + CSS as separate files from Pico W
# ── Multi-file web server — serves dashboard.html and styles.css ──
# Save both files to the Pico W root using Thonny → File → Save as → MicroPython device
import socket, time

def serve_file(conn, path):
    content_types = {
        '.html': 'text/html',
        '.css':  'text/css',
        '.js':   'application/javascript',
    }
    ext = path[path.rfind('.'):]  if '.' in path else ''
    ctype = content_types.get(ext, 'text/plain')
    try:
        with open(path, 'r') as f:
            body = f.read()
        conn.sendall(('HTTP/1.1 200 OK\r\nContent-Type: '
                      + ctype + '\r\nConnection: close\r\n\r\n' + body).encode())
    except OSError:
        conn.sendall(b'HTTP/1.1 404 Not Found\r\n\r\nFile not found')

srv = socket.socket()
srv.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)
srv.bind(('', 80))
srv.listen(3)
print('Multi-file server ready')

while True:
    try:
        conn, _ = srv.accept()
        req = conn.recv(1024).decode()
        # Extract the requested path from the GET line
        path = req.split(' ')[1] if ' ' in req else '/'
        if path == '/' or path == '/index.html':
            serve_file(conn, 'dashboard.html')
        elif path == '/styles.css':
            serve_file(conn, 'styles.css')
        else:
            conn.sendall(b'HTTP/1.1 404 Not Found\r\n\r\nNot found')
        conn.close()
    except Exception as e:
        print('Error:', e)
        try: conn.close()
        except: pass
