Student Scaffold

FIA2 Scaffold — Unit 1 Creating with Code

A student-friendly planning and writing tool aligned to the Digital Solutions syllabus and task expectations.

How to use this scaffold

This scaffold helps you build your own original FIA2 response. It is not a script to copy. Use it to plan what to write, what evidence to include, and how to explain your decisions clearly.

Important: Do not copy from exemplars. Your goal is to create an original response using your own system, your own wording and your own testing evidence.

Part A — Understand the problem

Answer these questions:
  • Who is the user?
  • What problem are they experiencing?
  • Why does the problem matter?
  • What inputs, processes and outputs will the system need?
  • What constraints must you consider?

Sentence starter: “The user needs a digital solution because … This problem is significant because …”

Sentence starter: “The problem can be decomposed into sensing, processing, communication, output and reliability.”

Part B — Explore and justify design ideas

Briefly compare existing products, interfaces or design patterns. You are not just listing them — you are showing how they influenced your design.

Example promptWhat to write
Existing solutionWhat it does well and what it does poorly for your user
Design implicationWhat you will copy, avoid or improve in your own solution
Emerging technologyA relevant future technology and how it could improve the system
Tip: Strong responses explain why each explored idea matters for the user, not just whether the product is “good” or “bad”.

Part C — Write strong success criteria

Your success criteria should be specific, measurable and directly connected to the user’s needs.

CriterionWhy it mattersHow you will test it
Status is shown with text and symbolImproves accessibilityCheck dashboard in greyscale mode
Dashboard refreshes automaticallySupports passive useMeasure refresh timing with stopwatch
No raw values in main displayReduces cognitive loadVisual inspection of screen
Sentence starter: “A successful solution must … because the user … This will be verified by …”
Avoid: vague statements like “The solution should work well” or “The display should be easy to use.” Be precise.

Part D — Explain your solution architecture

Describe how your system is organised. Explain what each component does and why this structure is suitable.

Include:
  • What each device/component does
  • How data moves through the system
  • Why the design supports maintainability, usability or reliability

Sentence starter: “The solution uses a two-part architecture. The first component is responsible for … while the second component …”

Part E — Pseudocode and algorithm thinking

Your pseudocode should show the key logic clearly. It should not just exist — you should also explain why the logic is structured that way.

Make sure you show:
  • assignment
  • sequence
  • selection
  • iteration
  • modularisation
  • conditions

Sentence starter: “The classification function is modularised so that …”

Sentence starter: “The selection logic gives priority to … because …”

Part F — Code explanation

When you include prototype code, explain what the important parts do. Focus on why the code matters, not just what line comes next.

Useful things to explain:
  • constants and thresholds
  • functions and their purpose
  • how classification works
  • how data is transmitted
  • how offline or error states are handled

Part G — Testing and evaluation

Strong evaluation includes both success and problems. If something partially fails, that can actually strengthen your response if you explain the refinement clearly.

TestExpected resultActual resultWhat this tells you
Boundary value testExact threshold behaves correctlyRecord your resultDoes your logic handle edge cases?
Live hardware testDevice responds as plannedRecord your resultDoes the real system match the design?
Offline/error testWarning appears in timeRecord your resultIs the system reliable?
Sentence starter: “This criterion was only partially met because … To improve this, I refined … and retested by …”

Part H — Impacts and recommendations

Evaluate the wider effects of your solution, not just whether the code works.

Consider:
  • Personal: How does the solution affect the user directly?
  • Social: Are there privacy, accessibility or safety issues?
  • Economic: Is the system affordable, efficient or scalable?

Sentence starter: “A positive personal impact is … However, a possible risk is …”

Sentence starter: “A justified recommendation for future improvement is … because testing showed …”

Rubric-aligned checklist

  • I explained the user and the problem clearly.
  • I decomposed the problem into smaller parts.
  • I wrote measurable success criteria.
  • I justified my architecture and algorithm choices.
  • I showed the required pseudocode constructs.
  • I explained important code sections.
  • I used testing evidence, not just opinions.
  • I included at least one meaningful refinement.
  • I discussed personal, social and economic impacts.
  • I used clear, formal and original communication.

Common mistakes to avoid

  • Writing criteria that are too vague to test
  • Describing code without explaining design choices
  • Only including passing tests and no reflection
  • Giving recommendations that are not linked to evidence
  • Copying exemplar wording too closely
