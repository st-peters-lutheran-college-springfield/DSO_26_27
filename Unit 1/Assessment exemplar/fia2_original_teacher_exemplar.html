Purpose of this resource

This exemplar is intended to model strong analytical and evaluative thinking without being copied directly by students. It keeps the strongest features of the uploaded exemplar — measurable criteria, clear justification, purposeful pseudocode, testing evidence and justified refinements — while changing the scenario, structure, phrasing and examples.

Scenario

User: A Year 11 student-athlete, Amira, needs a bedroom environment monitor that helps her prepare for recovery after training. She needs a simple display showing whether her room conditions are appropriate for rest and study recovery. The system should reduce cognitive load, avoid depending only on colour, and notify her if sensor data is no longer current.

Problem framing and decomposition

  • Input: temperature, light and motion readings from sensors
  • Processing: data must be classified into meaningful room-state categories
  • Output: a browser-based dashboard with a symbol and label
  • Transmission: wireless sending from sensor node to display server
  • Reliability: stale data must trigger an explicit warning
Why this is strong: Rather than retelling the whole story, this section breaks the problem into computable parts and shows systems thinking.

Solution concept and architecture

The prototype uses a distributed design. An ESP8266 collects environmental data and applies classification rules. An ESP32 hosts the web interface and receives data via HTTP. This separation improves maintainability because the sensing logic and display logic can be refined independently.

Annotated pseudocode

FUNCTION classifyEnvironment(temp, light, motion)
  DECLARE temp_ok, light_ok, motion_ok AS BOOLEAN

  temp_ok = temp >= 18 AND temp <= 22
  light_ok = light < 200
  motion_ok = motion = 0

  IF temp_ok AND light_ok AND motion_ok THEN
    RETURN "READY"
  ELSE IF NOT temp_ok OR NOT light_ok THEN
    RETURN "NOT_READY"
  ELSE
    RETURN "CAUTION"
  END IF
END FUNCTION

FUNCTION main()
  connectWiFi()
  WHILE TRUE
    temp, light, motion = readSensors()
    status = classifyEnvironment(temp, light, motion)
    sendData(status)
    wait(5)
  END WHILE
END FUNCTION
Teacher note: This keeps explicit algorithmic logic, but is presented more compactly than the uploaded exemplar so it reads as a different model rather than a close paraphrase.

A meaningful refinement followed the partial result in the offline test. The stale-data constant was reduced to account for network delay, then the system was retested. The second trial produced the warning in 13 seconds, which better aligned with the criterion.

Impacts and recommendations

Personal: reduces effort needed to judge room readiness.

Social: accessible design principles support broader usability.

Economic: the prototype uses low-cost hardware compared with commercial systems.

Recommendation: extend the system to include humidity because live testing may show conditions that feel uncomfortable even when temperature alone is acceptable.

How teachers can use this

  • Model how to justify success criteria with measurable thresholds
  • Show how evaluation should include partial failures and refinements
  • Demonstrate that strong responses explain why a design choice was made
  • Use as a quality reference, not as a script for student copying
