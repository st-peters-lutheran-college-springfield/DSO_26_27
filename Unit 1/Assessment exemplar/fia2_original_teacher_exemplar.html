Purpose of this resource

This exemplar models strong analytical and evaluative thinking aligned to the Digital Solutions syllabus. It keeps the strongest features of the earlier uploaded exemplar — measurable criteria, clear justification, purposeful algorithm design, testing evidence and meaningful refinement — while changing the user context, wording and overall structure.

Teaching focus: Use this page to highlight what makes a response strong: precise criteria, explicit logic, evidence-based evaluation and clear communication.

Scenario

User: Amira is a Year 11 student-athlete who needs a simple bedroom recovery monitor after training. She wants a quick room-readiness display that helps her decide whether conditions are suitable for rest and study recovery without needing to interpret raw sensor data.

Problem framing and decomposition

  • Input: collect temperature, light and motion data
  • Processing: classify raw data into a meaningful room-state category
  • Output: present a clear browser-based status with text and symbol
  • Transmission: move data wirelessly from the sensing device to the display server
  • Reliability: show a warning when information is stale or unavailable
Why this works in class: This section shows students how to break a real-world problem into computable system components instead of retelling the persona story.

Solution concept and architecture

The prototype uses a distributed design. An ESP8266 collects environmental readings and applies threshold logic. An ESP32 hosts the web interface and receives updates through HTTP. This structure improves maintainability because sensing logic and display logic can be adjusted independently.

Annotated pseudocode

FUNCTION classifyEnvironment(temp, light, motion)
  DECLARE temp_ok, light_ok, motion_ok AS BOOLEAN

  temp_ok = temp >= 18 AND temp <= 22
  light_ok = light < 200
  motion_ok = motion = 0

  IF temp_ok AND light_ok AND motion_ok THEN
    RETURN "READY"
  ELSE IF NOT temp_ok OR NOT light_ok THEN
    RETURN "NOT_READY"
  ELSE
    RETURN "CAUTION"
  END IF
END FUNCTION

FUNCTION main()
  connectWiFi()
  WHILE TRUE
    temp, light, motion = readSensors()
    status = classifyEnvironment(temp, light, motion)
    sendData(status)
    wait(5)
  END WHILE
END FUNCTION
Teacher note: This keeps algorithmic thinking explicit, but the shorter layout and alternate wording help separate it from the earlier exemplar students may have seen.

A meaningful refinement followed the partial result in the offline test. The stale-data constant was reduced to account for network delay, then the system was retested. The second trial produced the warning in 13 seconds, which better aligned with the criterion.

Impacts and recommendations

Personal: the system reduces effort needed to judge room readiness.

Social: accessible design features support broader usability.

Economic: the prototype uses low-cost hardware compared with commercial monitoring systems.

Recommendation: extend the system to include humidity because live testing may show conditions that feel uncomfortable even when temperature alone appears acceptable.

How teachers can use this

  • Show how evaluation should include partial failure and refinement
