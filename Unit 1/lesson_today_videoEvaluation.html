📓 Journal Task 1 — ~10 minutes

Open your OneNote journal. Pick any app or website you use every day (e.g. your school portal, a game, YouTube, Spotify). Write 3–5 sentences about its user experience. Use these dot-point words: accessibility, effectiveness, utility, learnability. Is it a good or poor UX? Give a specific example.

Part 2 — The five useability principles & Daniel's dashboard

♿ Accessibility

Can everyone use the interface, including people with disabilities or impairments?

→ Daniel has colour-vision deficiency. His dashboard uses ✓/● symbols and text labels — not colour alone.

✅ Effectiveness

Does the interface help users achieve their goal accurately and completely?

→ Daniel's goal: know if room is calm. Dashboard shows "ROOM IS CALM" — one glance, goal achieved.

🛡️ Safety

Does the design prevent user errors and protect them from dangerous mistakes?

→ Dashboard is read-only — Daniel can't accidentally change settings when half-asleep.

🔧 Utility

Does the interface provide the right functions for what the user actually needs?

→ Dashboard shows motion status and calm timer — exactly what Daniel needs, nothing more.

📖 Learnability

How quickly can a new user understand how to use the interface?

→ No buttons, no menus, no login. Daniel can use it with zero instruction — maximum learnability.
📓 Journal Task 2 — ~8 minutes

In your journal, write one sentence for each of the five useability principles explaining how your PIR dashboard either meets or fails to meet that principle for Daniel. Then write one improvement idea that would better satisfy one of the principles.

Part 3 — Visual design principles (CARP) in the QCAA syllabus

The QCAA syllabus lists these visual design principles for Unit 1 Topic 2: balance, contrast, proximity, harmony, alignment, repetition and hierarchy. The acronym CARP (Contrast, Alignment, Repetition, Proximity) comes from Robin Williams' The Non-Designer's Design Book and covers four of the most important ones for screen-based interfaces — it's a useful memory tool for the principles the syllabus requires.

C — Contrast

Make different things look different. If two things aren't the same, make them very different — not just slightly different.

  • Dark background (#111) vs near-white text (#eee)
  • Large symbol (✓) vs small label text
  • Solid / dashed / dotted borders to signal state

Syllabus link: contrast, hierarchy, tone

A — Alignment

Every element should have a visual connection to something else on the page. Nothing should be placed arbitrarily.

  • All text centred — consistent axis throughout
  • Reading tiles aligned in a grid
  • Progress bar full-width — edge-to-edge alignment

Syllabus link: alignment, balance

R — Repetition

Repeat visual elements throughout the design. This creates consistency and strengthens unity.

  • Same monospace font everywhere
  • Same tile style for every reading card
  • Same border-radius on all containers

Syllabus link: repetition, harmony

P — Proximity

Group related items together. Items that are related should be close; unrelated items should be separated.

  • Symbol + status label + message grouped in one card
  • Progress bar separated from status card
  • Reading tiles separate from main status area

Syllabus link: proximity, hierarchy

🎨 Remaining syllabus principles — also in your design

Balance

Visual weight is distributed evenly. Daniel's dashboard is vertically centred and full-width — no heavy left or right side that feels lopsided.

Harmony

All visual elements feel like they belong together. Using a single font family, monochrome palette, and consistent spacing creates visual harmony.

Hierarchy

The most important information (status symbol + label) is largest and at the top. Secondary data (readings) is smaller below. Daniel's eye goes to the right thing first.

Part 4 — Your 2-minute assessment video

🎬 What the assessment requires

From the FIA2 task sheet: "Visual and spoken video: up to 2 minutes, demonstrating the functionality of the user interface and data and programmed components." This is a Communicating criterion task (Objective 8, 2 marks).

✅ Must include

  • Your dashboard running live in a browser
  • At least one sensor changing the status (wave hand at PIR)
  • Your spoken explanation of what is happening
  • Reference to Daniel's needs and your design decisions
  • At least one UX or visual design principle named and explained

🚫 Do not include

  • More than 2 minutes of footage
  • Just showing code — the video must show the running interface
  • Silent screen recording — you must speak
  • Background music that drowns out your voice
  • Other students' names or faces

📁 Format requirements

  • mp4 file format
  • Low resolution, small file size
  • Record your screen + microphone
  • Windows: Xbox Game Bar (Win+G)
  • Mac: QuickTime Player → New Screen Recording
  • Or: OBS Studio (free, any OS)

SECTION
WHAT TO SAY AND SHOW

Functionality
Show: Wave your hand at the PIR — dashboard changes to MOTION DETECTED ●.
Say: "When motion is detected, the status changes immediately. The PIR sensor detects heat movement and updates the dashboard automatically every 5 seconds — Daniel never has to press anything."
  • Show all three states if possible: DETECTED → SETTLING ~ → CALM ✓
  • Point out the progress bar filling as the room settles
0:50 – 1:20
UX Design
Show: Point to specific elements of the dashboard.
Say: "My design uses contrast — dark background with near-white text — so it's readable in any lighting. I used symbols alongside text because Daniel has mild colour-vision deficiency, which means I can't rely on colour alone. This addresses the accessibility useability principle."
  • Name at least one more principle: e.g. "I used proximity to group related information…"
  • Mention border style changes (solid/dashed/dotted) as an accessibility decision
1:20 – 1:45
Code
Show: Briefly switch to Thonny — show the get_status() function OR the while True server loop.
Say: "The algorithm uses an if/elif/else selection statement to decide which status to display based on how long ago motion was last detected. The server uses a while True iteration loop to continuously serve the page to any browser on the network."
  • Name selection + iteration — these are required programming concepts
  • Don't read all the code — just identify the key concept
1:45 – 2:00
Wrap-up
Show: Dashboard on final CALM ✓ state.
Say: "This starting-point solution meets Daniel's core need — a clear, accessible, live status with no interaction required. My next step is to add temperature and humidity sensors to give a more complete rest-suitability assessment."
📓 Journal Task 4 — ~10 minutes — video plan

In your OneNote journal, write your own version of the 2-minute script above. Personalise it: use specific things from your code and your dashboard. You don't need to memorise it — having the plan means your recording will be focused and cover all the required elements. Aim for about 250–300 words spoken at a natural pace.
